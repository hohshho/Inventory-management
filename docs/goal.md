# 재고관리 시스템 기능 요구사항

## 1. 문서 목적
- 본 문서는 재고관리 시스템의 주요 기능 요구사항을 정의한다.
- 시스템은 일반 재고관리뿐 아니라 냉장고 재고관리와 같은 사용 사례도 지원해야 한다.

## 2. 대상 사용자
- 주요 사용자는 채위생사 등 현장 실무자이다.
- 시스템은 비전문가도 쉽게 사용할 수 있도록 직관적인 흐름으로 설계해야 한다.

## 3. 기능 요구사항

### 3.1 재고관리 지원
- 시스템은 일반 재고 품목을 등록, 조회, 수정, 삭제할 수 있어야 한다.
- 시스템은 냉장 또는 보관 위치 기반의 재고관리 시나리오를 지원할 수 있어야 한다.

### 3.2 바코드 기반 작업
- 사용자는 바코드를 스캔하여 품목을 식별할 수 있어야 한다.
- 바코드 스캔 시 시스템은 해당 품목의 재고 등록 또는 생성 페이지로 자동 이동해야 한다.
- 바코드가 기존 품목과 연결되지 않은 경우, 신규 품목 등록 흐름으로 이어질 수 있어야 한다.

### 3.3 사용 편의성
- 사용자는 최소한의 입력만으로 재고를 등록할 수 있어야 한다.
- 현장 사용 환경을 고려하여 빠른 입력과 단순한 화면 흐름을 제공해야 한다.
- 주요 작업은 학습 부담이 낮은 UI 구조로 제공해야 한다.

## 4. UI/UX 요구사항
- UI는 BoxHero 스타일을 참고하여 구성한다.
- 화면은 재고 조회, 등록, 수정, 이동 등의 핵심 작업이 빠르게 이루어지도록 설계해야 한다.
- 복잡한 관리 화면보다 현장 중심의 단순하고 명확한 인터페이스를 우선한다.

## 5. 기능 도출 기준
- 세부 기능 기획은 BoxHero 공식 문서(https://docs-ko.boxhero.io/)를 참고하여 추출한다.
- 필요 시 위 문서를 기준으로 추가 기능 목록과 상세 정책을 확장 정의한다.

## 6. MVP 범위

### 6.1 MVP에 포함할 기능
- 품목 등록, 조회, 수정, 삭제 기능을 제공해야 한다.
- 현재 재고 수량을 확인하고 수량을 직접 조정할 수 있어야 한다.
- 보관 위치 또는 냉장고 구역 기준으로 품목을 구분해 관리할 수 있어야 한다.
- 바코드 스캔을 통해 기존 품목 조회 또는 신규 품목 등록으로 연결할 수 있어야 한다.
- 모바일 또는 태블릿 환경에서도 현장 사용이 가능해야 한다.
- 직관적인 대시보드 또는 목록 화면에서 재고 현황을 빠르게 파악할 수 있어야 한다.

### 6.2 MVP 제외 또는 후순위 기능
- 고급 통계 리포트 및 분석 기능
- 외부 ERP, 회계, 구매 시스템 연동
- 다국어 지원
- 복잡한 권한 체계 및 조직 단위별 세부 권한 제어
- 자동 발주, 자동 정산 등 확장 운영 기능

## 7. 추후 확장 범위
- 유통기한 관리 및 만료 예정 품목 알림 기능
- 입고, 출고, 폐기 이력 관리 기능
- 재고 부족 알림 및 기준 수량 설정 기능
- 다수 창고 또는 다수 지점 관리 기능
- 사용자별 작업 로그 및 감사 이력 기능
- CSV 또는 엑셀 업로드/다운로드 기능

## 8. 핵심 사용자 스토리

### 8.1 재고 등록
- 현장 사용자는 새로운 품목을 빠르게 등록하여 즉시 재고관리를 시작할 수 있어야 한다.
- 현장 사용자는 품목명, 수량, 보관 위치 등 최소한의 정보만 입력해 등록할 수 있어야 한다.

### 8.2 바코드 스캔
- 현장 사용자는 바코드를 스캔해 기존 품목을 바로 찾고 재고를 갱신할 수 있어야 한다.
- 현장 사용자는 미등록 바코드를 스캔한 경우 신규 품목 등록 화면으로 자연스럽게 이어질 수 있어야 한다.

### 8.3 재고 조회
- 현장 사용자는 현재 재고 현황을 목록 형태로 빠르게 확인할 수 있어야 한다.
- 현장 사용자는 품목명, 위치, 바코드 등으로 원하는 품목을 검색할 수 있어야 한다.

### 8.4 위치 기반 관리
- 현장 사용자는 냉장고, 창고, 선반 등 보관 위치별로 재고를 구분해 관리할 수 있어야 한다.
- 현장 사용자는 특정 위치의 재고만 모아서 확인할 수 있어야 한다.

## 9. 인수조건

### 9.1 품목 등록
- 사용자가 필수 정보를 입력하고 저장하면 신규 품목이 목록에 즉시 반영되어야 한다.
- 필수 입력값이 누락된 경우 저장되지 않아야 하며, 누락 항목을 명확히 안내해야 한다.

### 9.2 바코드 스캔
- 등록된 바코드를 스캔하면 해당 품목 상세 또는 재고 처리 화면으로 이동해야 한다.
- 등록되지 않은 바코드를 스캔하면 신규 등록 화면으로 이동해야 한다.
- 스캔 실패 또는 인식 불가 시 사용자가 다시 시도할 수 있도록 안내해야 한다.

### 9.3 재고 수량 변경
- 사용자가 재고 수량을 수정하면 변경된 수량이 즉시 반영되어야 한다.
- 수량 변경 시 음수 입력 또는 비정상 입력은 제한되어야 한다.

### 9.4 조회 및 검색
- 사용자가 품목명 또는 바코드로 검색하면 관련 품목만 필터링되어 보여야 한다.
- 사용자가 위치를 선택하면 해당 위치의 품목만 조회할 수 있어야 한다.

### 9.5 사용성
- 주요 기능은 3단계 이내의 화면 흐름으로 수행 가능해야 한다.
- 모바일 화면에서도 주요 버튼과 입력 요소를 무리 없이 사용할 수 있어야 한다.

## 10. 화면 목록

### 10.1 대시보드
- 목적: 전체 재고 현황과 주요 작업 진입점을 빠르게 제공한다.
- 주요 정보: 총 품목 수, 부족 재고 수, 위치별 재고 요약, 최근 수정 내역
- 주요 액션: 품목 등록, 바코드 스캔, 재고 조회 화면 이동
- 우선순위: MVP

### 10.2 재고 목록 화면
- 목적: 전체 재고를 조회하고 검색, 필터, 수정 진입을 제공한다.
- 주요 정보: 품목명, 바코드, 현재 수량, 단위, 위치, 최종 수정일
- 주요 액션: 검색, 위치 필터, 품목 상세 이동, 신규 등록
- 우선순위: MVP

### 10.3 품목 등록/수정 화면
- 목적: 품목 정보를 등록하거나 수정한다.
- 주요 정보: 품목명, 바코드, 보관 위치, 현재 수량, 단위, 메모
- 주요 액션: 저장, 취소, 삭제, 바코드 직접 입력 또는 스캔 연동
- 우선순위: MVP

### 10.4 품목 상세 화면
- 목적: 특정 품목의 상세 정보와 재고 상태를 확인한다.
- 주요 정보: 품목 기본 정보, 현재 수량, 위치, 최근 변경 이력
- 주요 액션: 재고 수량 조정, 수정 화면 이동, 삭제
- 우선순위: MVP

### 10.5 바코드 스캔 화면
- 목적: 바코드 인식을 통해 품목 조회 또는 신규 등록 흐름으로 연결한다.
- 주요 정보: 카메라 스캔 영역, 최근 스캔 결과, 인식 실패 안내
- 주요 액션: 스캔 시작, 수동 입력 전환, 등록 화면 이동, 상세 화면 이동
- 우선순위: MVP

### 10.6 위치 관리 화면
- 목적: 냉장고, 창고, 선반 등 보관 위치를 등록하고 관리한다.
- 주요 정보: 위치명, 위치 유형, 설명, 등록된 품목 수
- 주요 액션: 위치 등록, 수정, 삭제, 해당 위치 재고 조회
- 우선순위: MVP

### 10.7 재고 변경 이력 화면
- 목적: 입고, 수정, 차감 등의 재고 변경 기록을 확인한다.
- 주요 정보: 품목명, 변경 전 수량, 변경 후 수량, 변경 사유, 작업자, 일시
- 주요 액션: 기간 조회, 품목별 필터, 위치별 필터
- 우선순위: 추후 확장 또는 MVP 후반

## 11. 화면별 기능 명세

### 11.1 대시보드
- 시스템은 로그인 직후 또는 앱 진입 직후 핵심 현황을 요약해 보여줘야 한다.
- 시스템은 최근 수정된 품목과 자주 사용하는 메뉴를 상단에 배치해야 한다.
- 시스템은 모바일 환경에서 주요 CTA 버튼이 첫 화면 내에 보이도록 구성해야 한다.

### 11.2 재고 목록 화면
- 시스템은 품목 목록을 페이지네이션 또는 무한 스크롤 방식으로 제공할 수 있어야 한다.
- 시스템은 품목명, 바코드, 위치 기준 검색 및 필터를 지원해야 한다.
- 시스템은 목록에서 현재 수량과 위치를 즉시 확인할 수 있어야 한다.
- 시스템은 목록에서 품목 상세 또는 수정 화면으로 빠르게 이동할 수 있어야 한다.

### 11.3 품목 등록/수정 화면
- 시스템은 필수 입력값과 선택 입력값을 명확히 구분해야 한다.
- 시스템은 등록 시 기본 수량과 위치를 함께 입력할 수 있어야 한다.
- 시스템은 수정 시 기존 값을 불러와 편집 가능해야 한다.
- 시스템은 저장 성공 및 실패 결과를 사용자에게 즉시 안내해야 한다.

### 11.4 품목 상세 화면
- 시스템은 품목의 현재 상태를 한 화면에서 확인할 수 있어야 한다.
- 시스템은 재고 수량 조정 기능을 상세 화면에서 직접 제공해야 한다.
- 시스템은 최근 변경 이력을 통해 누가 언제 수량을 수정했는지 확인할 수 있어야 한다.

### 11.5 바코드 스캔 화면
- 시스템은 카메라 권한이 없는 경우 권한 요청 또는 대체 입력 수단을 제공해야 한다.
- 시스템은 스캔 성공 시 기존 품목 여부를 판별하여 적절한 화면으로 이동해야 한다.
- 시스템은 연속 스캔 환경을 고려해 빠른 복귀 또는 재스캔 기능을 제공해야 한다.

### 11.6 위치 관리 화면
- 시스템은 위치명을 고유하게 관리해야 한다.
- 시스템은 위치 삭제 시 연결된 품목 존재 여부를 검증해야 한다.
- 시스템은 위치별 등록 품목 수를 확인할 수 있어야 한다.

## 13. API 초안

### 13.1 품목 API
- `GET /api/items`: 품목 목록 조회
- `POST /api/items`: 신규 품목 등록
- `GET /api/items/{itemId}`: 품목 상세 조회
- `PATCH /api/items/{itemId}`: 품목 정보 수정
- `DELETE /api/items/{itemId}`: 품목 삭제

### 13.2 위치 API
- `GET /api/locations`: 위치 목록 조회
- `POST /api/locations`: 위치 등록
- `PATCH /api/locations/{locationId}`: 위치 수정
- `DELETE /api/locations/{locationId}`: 위치 삭제

### 13.3 바코드 API
- `POST /api/barcodes/resolve`: 바코드 스캔 결과로 기존 품목 여부 판별
- `POST /api/barcodes`: 품목에 바코드 연결

### 13.4 재고 API
- `POST /api/inventory-adjustments`: 재고 수량 조정
- `GET /api/inventory-adjustments`: 재고 변경 이력 조회
- `GET /api/dashboard/summary`: 대시보드 요약 정보 조회

### 13.5 API 응답 원칙
- 모든 API는 성공 여부, 메시지, 데이터 구조를 일관되게 반환해야 한다.
- 유효성 검증 실패 시 필드 단위 오류 메시지를 반환해야 한다.
- 모바일 환경을 고려해 불필요하게 큰 응답은 피해야 한다.

## 14. DB 초안

### 14.1 items
- 목적: 품목 기본 정보를 저장한다.
- 주요 컬럼: `id`, `name`, `default_unit`, `memo`, `is_active`, `created_at`, `updated_at`

### 14.2 item_barcodes
- 목적: 품목과 바코드 매핑 정보를 저장한다.
- 주요 컬럼: `id`, `item_id`, `barcode`, `created_at`
- 제약 조건: `barcode`는 고유값이어야 한다.

### 14.3 locations
- 목적: 냉장고, 창고, 선반 등 보관 위치 정보를 저장한다.
- 주요 컬럼: `id`, `name`, `type`, `description`, `created_at`, `updated_at`
- 제약 조건: `name`은 고유값이어야 한다.

### 14.4 inventories
- 목적: 품목별 현재 재고 수량과 위치 정보를 저장한다.
- 주요 컬럼: `id`, `item_id`, `location_id`, `quantity`, `updated_at`
- 제약 조건: `item_id`와 `location_id` 조합은 중복되지 않아야 한다.

### 14.5 inventory_adjustments
- 목적: 재고 수량 변경 이력을 저장한다.
- 주요 컬럼: `id`, `item_id`, `location_id`, `before_quantity`, `after_quantity`, `change_type`, `reason`, `created_by`, `created_at`

### 14.6 users
- 목적: 작업자 또는 관리자 정보를 저장한다.
- 주요 컬럼: `id`, `name`, `email`, `role`, `created_at`, `updated_at`
- 비고: 인증 기능이 늦게 들어가더라도 변경 이력 기록용 최소 사용자 구조는 고려하는 것이 좋다.

## 15. 개발 우선순위 제안
- 1순위: 품목 CRUD, 위치 관리, 재고 목록, 재고 수량 조정
- 2순위: 바코드 스캔 연동, 대시보드 요약, 모바일 최적화
- 3순위: 재고 변경 이력 조회, 알림, 유통기한 관리
